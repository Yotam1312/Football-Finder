Edit `backend/prisma/schema.prisma` to make three changes in a single edit:

**1. Stadium model — add 6 transport fields after the `matches` relation (before the `@@index`):**

```prisma
  // Transport fields (Phase 15) — populated per-stadium by admin via CSV/SQL
  nearbyMetros          String[]   // e.g. ["Line 1 - Stade de France", "RER D"]
  nearbyTrains          String[]   // e.g. ["RER B - Paris Nord"]
  nearbyBuses           String[]   // e.g. ["Bus 154", "Bus 255"]
  walkingTimeFromCenter String?    // e.g. "25 min walk from Gare du Nord"
  publicTransportInfo   String?    // Free-text general transport tip
  parkingInfo           String?    // Free-text parking notes
```

`String[]` maps to PostgreSQL `TEXT[] NOT NULL DEFAULT ARRAY[]::TEXT[]` — existing rows get empty arrays automatically.
`String?` maps to nullable `TEXT` — existing rows get NULL automatically.

**2. PostType enum — add `GETTING_THERE` as the last value:**

```prisma
enum PostType {
  GENERAL_TIP
  SEAT_TIP
  PUB_RECOMMENDATION
  IM_GOING
  GETTING_THERE   // Added in Phase 15 for community transport tips
}
```

**3. Post model — add 3 optional transport fields after the `upvotes` relation (before the `@@index` lines):**

```prisma
  // Transport tip fields (Phase 15) — only populated when postType = GETTING_THERE
  transportType String?   // "Metro" | "Bus" | "Train" | "Taxi" | "Walking" | "Other"
  travelCost    String?   // e.g. "€1.90 per trip"
  travelTime    String?   // e.g. "20 min from city centre"
```

All three are `String?` (nullable text). `transportType` is intentionally a plain string, not an enum — Phase 18 enforces allowed values at the application layer.
